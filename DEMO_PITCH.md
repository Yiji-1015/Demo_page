# 신입사원 온보딩 봇 — 데모 시연 가이드

## 핵심 포지셔닝 메시지

> "단순한 Q&A 챗봇이 아닌, **회사 지식이 살아있는 RAG 표준 파이프라인**입니다."

---

## 차별화 포인트

### 1. Confluence 직연동 — "문서를 다시 만들 필요가 없습니다"
- 기존 Confluence에 올린 문서를 그대로 활용
- 사이드바에서 원하는 문서를 선택해 임베딩하면 즉시 지식베이스에 추가
- *"HR이 새 문서를 올리면, 담당자가 동기화 버튼을 눌러 챗봇에 반영합니다"*

> **주의**: Confluence 원문이 수정되어도 자동으로 반영되지 않습니다.
> 변경 사항을 반영하려면 **해당 문서를 다시 선택해 재임베딩**해야 합니다.
> (Elasticsearch에 저장된 스냅샷 기준으로 답변)

---

### 2. 표(Table) 전용 파싱 로직 — "구조화된 정보도 제대로 읽습니다"
일반 텍스트 추출만 하면 표는 데이터가 뭉개집니다. 이 시스템은 **표를 별도 로직으로 처리**합니다.

- `rowspan` / `colspan` 병합 셀을 펼쳐서 **그리드로 평탄화**
- 표를 **Markdown Table 형식**으로 변환 후 텍스트와 별도 블록으로 저장
- 최종 저장 구조: `본문 텍스트` + `[표]` + `[하위 페이지 목록]` 순으로 조합
- *"복잡한 인사 규정표나 복리후생 비교표도 데이터 손실 없이 검색에 반영됩니다"*

---

### 3. Confluence 전용 요소 처리 — "위키 문서에 특화된 파싱"
Confluence HTML은 일반 HTML과 다른 전용 태그를 사용합니다.

- `ac:link` 태그 → `[텍스트](관련문서: 제목)` 형태로 변환해 관계 문서 맥락 보존
- `ri:attachment` → 첨부파일명 추출
- `ac:structured-macro children` → 하위 페이지 목록 자동 포함
- `script`, `style`, `ac:parameter` 등 노이즈 태그 제거
- *"Confluence 특유의 매크로나 링크 구조도 의미 있는 텍스트로 변환합니다"*

---

### 4. 하이브리드 검색 (Vector + BM25) — "한국어에 강합니다"
- **Nori 한국어 형태소 분석기** + 벡터 검색을 **3:2 비율**로 결합
- 제목 필드는 본문보다 2배 가중치(`title^2`) 부여
- 검색 결과 5개를 LLM에 모두 전달 → 사용자에게는 상위 3개만 노출
- *"'육아휴직 신청 절차'처럼 실무 키워드로 물어봐도 정확한 문서를 찾습니다"*

---

### 5. 인하우스 임베딩 모델 (BGE-M3, 1024차원) — "외부 API에 데이터가 안 나갑니다"
- 문서 임베딩은 **사내 서버에서 처리**
- OpenAI에는 질의응답 최종 단계만 전달
- 청크 단위로 배치 전송(50개씩), 실패 시 개별 임베딩 폴백 처리
- *"민감한 사내 문서가 외부로 유출되지 않는 구조입니다"*

---

### 6. 청킹 시 제목 병합 — "어떤 청크를 읽어도 문서 맥락을 알 수 있습니다"
- 문서를 3000자 단위(200자 오버랩)로 분할
- 각 청크 앞에 **`[문서 제목: ...]`를 자동으로 붙여서** 저장
- *"긴 문서를 여러 조각으로 나눠도 LLM이 어느 문서의 내용인지 항상 파악합니다"*

---

### 7. SME(담당자) 자동 식별 — "이 문서는 누가 가장 잘 아는지 알 수 있습니다"
- Confluence 버전 히스토리 전체를 분석해 **최다 수정자**를 SME로 식별
- 수정 이력이 1건뿐이면 마지막 수정자를 담당자로 지정
- 검색 결과에 `primary_contributor` 필드로 함께 반환
- *"문서 내용이 불충분하면 바로 담당자에게 연락할 수 있습니다"*

---

### 8. Temperature = 0.0 & 출처 표시 — "신뢰할 수 있는 답변"
- 할루시네이션 최소화 설정
- 답변 하단에 **출처 문서 링크 + 최종 수정일** 자동 첨부
- *"챗봇이 틀린 말을 해도 원문을 바로 확인할 수 있습니다"*

---

### 9. 증분 임베딩 (중복 방지) — "운영 비용이 낮습니다"
- 이미 인덱싱된 문서는 재임베딩 하지 않음
- 변경된 문서만 선택적으로 업데이트 가능 (`force_update`)
- *"문서가 수백 개 늘어도 운영 부담이 선형적으로 증가하지 않습니다"*

---

## 데모 시연 흐름

```
1. 홈 대시보드         → "이런 AI 서비스들을 하나의 포털에서"
2. 온보딩 챗봇 진입    → 사이드바의 Confluence 문서 트리 보여주기
3. 실제 질문 시연      → "연차 사용 방법", "입사 후 처음에 뭘 해야 하나요?"
4. 답변 + 출처 링크    → 출처 투명성, 수정일, 담당자 정보 강조
5. 신규 문서 임베딩    → "선택 → 동기화, 이게 전부입니다"
```

## 자주 나오는 질문 대비

| 질문 | 답변 포인트 |
|------|------------|
| "GPT랑 뭐가 달라요?" | GPT는 회사 내부 문서를 모름. 우리 RAG는 사내 문서만 기반으로 답변 |
| "문서 업데이트는 어떻게 해요?" | Confluence 수정 후 해당 문서를 다시 선택해 재임베딩 필요 (자동 X) |
| "표가 있는 문서는요?" | rowspan/colspan 포함한 복잡한 표도 Markdown으로 변환해 검색에 반영 |
| "보안은?" | 임베딩은 사내, LLM 호출 시 문서 청크만 전달, 원문 전체 전송 안 함 |
| "틀린 답변이 나오면?" | 출처 링크로 원문 즉시 확인, 담당자(SME) 정보도 함께 제공 |
| "다른 시스템에도 쓸 수 있나요?" | Confluence 외 다른 소스로도 파이프라인 교체 가능한 구조 |
